# Working in `backend/`

FastAPI service for versioned project documents. `README.md` here has the run
commands and module layout; this file is the rules digest. Repo-wide rules and
the closeout gate are in `../CLAUDE.md`.

## Read before writing backend code

- `../context/CODING_STANDARDS.md` (Backend Python) — layer/feature boundaries,
  typing, module-size, documentation.
- `../context/TECH_STACK.md` — persistence and stack decisions, for any
  data-access or dependency question.

## Non-negotiables

- **`uv` only.** Never `python`/`python3`/`pip`/`source .venv/bin/activate`. Use
  `uv run <cmd>` / `uvx <tool>`; add deps with `uv add [--dev] <pkg>` (never
  hand-edit `uv.lock`). Python 3.11, **Pydantic v2 only**.
- **All calculations and data manipulation live here**, not in the frontend.
- **Every feature uses the same layers** under `features/<feature>/`:
  `routes.py` (FastAPI wiring only), `models.py` (Pydantic DTOs/enums),
  `service.py` (business logic), `repository.py` (raw parameterized SQL).
  Split large modules; document the *why* behind behavior.
- **Raw parameterized SQL through narrow repositories — no SQLAlchemy ORM/Core
  in app code.** Alembic may use SQLAlchemy internally, for migrations only;
  revisions are manual (no autogenerate).
- **Strictly typed** (`ty check`). No implicit `dict`/`list` contracts in
  service code — use Pydantic models or narrow Python types.
- `config.py` is the only `os.getenv` surface (Pydantic `Settings`). Feature
  flags are `Settings` fields with safe defaults — no `.env` overlays.

## Checks

`make check-backend`, or from `backend/`:
`uv run ty check && uv run ruff check . && uv run pytest`. The full gate before
reporting done is the closeout gate in `../CLAUDE.md`.

1. Executive summary

Phase 1 has landed on a solid skeleton: feature-first packages on both sides, raw-SQL repositories, Pydantic v2 boundaries, TanStack Query server-state ownership, working ETag/draft concurrency, an MCP read surface, and a single source-of-truth JSONB body. The build is healthy, the tests are real (not snapshot tar pits), and there is no obvious detour from the PRD's load-bearing decisions.

The principal friction is concentrated in one feature package: features/project_document/ has absorbed document schema, drafts, save/save-as/discard, version metadata patch, diff, and downloads. This is now a single ~525-line service module with two routers, and it draws the projects ↔ project_document dependency graph into a loop. Two more table types (Windows, Envelope) and this package becomes very hard to navigate. Splitting it now, while the only typed table is Rooms, costs almost nothing; doing it after TB-08/TB-10 will require moving a lot more code.

The other notable gap is on the frontend: there is no Zustand, no shadcn/Tailwind, no <DataTable>, no JSON Schema endpoint, and no idempotency-key handling — all of which the PRD/TECH_STACK calls out as v1 commitments. Each has been deferred deliberately per slice. Two of them (RoomsSliceReplaceRequest hardcoded shape, per-table BroadcastChannel hook) need a design call before Windows ships in TB-08, or the per-table pattern will be copy-pasted three times before anyone notices.

Verdict: the JSONB-body bet still pays off for v1. The plan is largely intact. Pause here, split project_document, generalize the table-slice contract, and pick the Zustand / styling cutover slice. Then proceed.

3. Primary review items

3.1 — Architectural friction emerging now ⚠ moderate

High A. project_document is becoming a god-feature

By the end of TB-06, backend/features/project_document/ has accumulated:

  • document schema (document.py);
  • Rooms-slice read/replace (service.py);
  • draft buffer upsert / delete (repository.py, service.py);
  • Save and Save As (service.py);
  • discard draft;
  • PATCH version metadata (lock / make-active);
  • version-vs-version and version-vs-draft diff;
  • project JSON + per-table JSON downloads (read-safe raw body recovery path).

service.py is now 525 lines — right at the CODING_STANDARDS "split before 600" soft limit — and routes.py mounts two routers (router + diff_router) just to avoid awkward prefix nesting. That dual-router file is a smell.

The PRD/API contract actually distinguishes:

  • §9.3 Versions (lifecycle: PATCH, DELETE, save-as-new POST)
  • §9.4 Document read
  • §9.5 Drafts (autosave / save / save-as / discard)
  • §9.6 Diff
  • §9.7 Downloads

These are five domains. Today they all live in one folder, behind one service module, exporting one set of helpers. The TB-05 code review already flagged this; it's worth resolving before TB-08 piles Windows-slice logic and TB-17 piles MCP writes on top.

Recommendation: Split into features/versions/ (PATCH version, save-as-new clone), features/drafts/ (save, save-as, discard, draft replace / patch), features/document/ (read endpoints + ProjectDocumentV1 + table-slice contract), features/diff/, and features/downloads/ (or merge diff+downloads if they stay thin). All five can share shared.errors.api_error and shared.audit.log_action.

High B. projects ↔ project_document circular dependency

From the import graph:

  • projects/repository.py imports ProjectDocumentV1 (for the empty initial version body)
  • projects/service.py imports ProjectDocumentProject, ProjectDocumentV1 (for empty_project_document)
  • project_document/document.py imports CertificationProgram from projects/models.py
  • project_document/models.py imports ProjectVersionPublic, VersionKind from projects/models.py
  • project_document/service.py imports ProjectDetail, body_size_bytes, get_project_detail, version_public from projects/service.py
  • project_document/routes.py imports ProjectDetail from projects/models.py

Python doesn't raise — there are no module-load cycles — but this is "package A and package B own halves of each other's contracts." The fix is the same as ADR-1: when project_document is split, document/version types belong with the document feature, and features/projects/ can stay narrow (project metadata, BT number availability, ownership, list/create/get).

Medium C. The Rooms slice shape will not survive Windows/Envelope

Today the table-slice contract is Rooms-shaped:

  • require_rooms_table(table_name) rejects every other table name with 404.
  • RoomsSliceReplaceRequest, RoomsSliceOptions, RoomsSliceResponse are concrete Pydantic classes per Rooms.
  • apply_rooms_replace, rooms_response are Rooms-specific helpers.
  • The frontend has parallel concrete RoomsSlice, RoomsReplacePayload types and a useRoomsSliceQuery, useRoomsDraftBroadcast pair.

This is fine as a tracer-bullet for TB-04 but it will become three or four near-copies the moment Windows (WindowTypesSliceReplaceRequest) and Envelope (AssembliesSliceReplaceRequest) arrive. Worth generalizing now while the shape is small.

Shape that would scale:

backend/features/document/tables/
  registry.py        # name → SliceContract { payload, response, replace_fn }
  rooms.py
  window_types.py    # added in TB-08
  assemblies.py      # added in TB-10

# routes:
PUT /draft/tables/{name}
  → look up SliceContract by name → validate → call replace_fn

On the frontend, the equivalent generalization is one useTableSliceQuery<T> and one useTableDraftBroadcast<T> parameterized on the table name + the "is this remote slice in conflict with my active edit" predicate.

Medium D. MCP server.py imports from every feature

This is by design (MCP wraps REST and reuses services), but features/mcp/server.py already imports from features/project_document, features/project_status, and features/projects. Each new MCP tool grows this cross-feature dependency list. Acceptable for v1; just keep the rule "MCP is a thin façade — it does not invent its own service workflow" explicit in CODING_STANDARDS or the MCP README, so the temptation to grow workflow logic in the MCP service file stays low.

Low E. ProjectDocumentV1.validate_document_references is hardcoded to Rooms

The model-level validator currently:

  • seeds rooms.floor_level and rooms.building_zone option lists if missing;
  • de-dupes option IDs and labels per option key;
  • walks tables.rooms looking for duplicate room IDs, duplicate room numbers, and references to missing option IDs.

This works because Rooms is the only typed table. When thermal_bridges.category, equipment.ervs.manufacturer, and friends arrive, this validator will fork into per-table branches. A small registry pattern (per-table option keys + per-table reference validator) keeps it manageable.

3.3 — JSONB single-body persistence: is it still the right move? ✓ yes for v1

Short answer: yes, keep it. The PRD's motivation (immutable per-version saves, file-format export, LLM-friendly whole-document fetch, no relational-shadow duality) is intact, and TB-04…TB-06 prove the model works on a real Rooms slice with ETag/draft/save flows. But the implementation has accrued some hot paths and coupling that should be designed out before tables multiply.

The pros are still real

  • One source of truth. No relational shadow of tables; no two-writes-per-edit problem.
  • Versions are trivially immutable. One UUID, one row, one byte-blob body.
  • Diff is just JSON-vs-JSON. No need to walk foreign keys across N tables.
  • Download = the body. Project JSON download is a single SELECT and a JSON serialize.
  • Bookshelf catalog model works naturally. catalog_origin sits inline on each copied entry; refresh-from-catalog is a deterministic field-level walk.
  • LLM-amenable. Whole-document fetch + stable IDs + schema-versioned + Pydantic-validated.

The cons that are showing up

  • O(whole document) Pydantic validation on every slice read. Every GET /draft/tables/rooms currently does ProjectDocumentV1.model_validate(version["body"]) on the entire body to return one table. At 50 KB it's negligible; at 5 MB it's a measurable tax on every keystroke-debounce. Performance review owns this — flag that the architecture permits a fast-path slice reader (skip validating untouched tables) and that the hot path is identifiable.
  • Single draft_etag per document, not per table. Same-editor "Rooms in tab A, Windows in tab B" works today because there's only Rooms. Once a second table ships, a Windows edit will bump the draft etag and the Rooms tab will see "draft changed elsewhere" — even though the change didn't touch its table. The TB-06 broadcast machinery partially papers over this for same-table-different-row, but not for cross-table. PRD §15 already names "per-table draft scoping if needed in v1.1" as a planned escape hatch.
  • Whole-document apply-and-validate on every table replace. apply_rooms_replace does body.model_copy(update={...}) on the whole document then revalidates. Acceptable. Same scalability ceiling as point 1.
  • No cross-project queries. Known, PRD accepts.

Verdict

Keep JSONB-as-source-of-truth. Do not add relational shadows of project tables in v1.

Two seams to design now to keep options open later:

  1. Slice contract registry (ADR-2) so each table type has its own payload + response + replace fn, and the route is generic.
  2. Lazy slice validation path in service code: when only one table is needed, validate that table's slice against its row-level model, not the whole document. The whole-document validator is still the authoritative gate on Save and Save As — those keep ProjectDocumentV1.model_validate(...).
3.4 — UI & frontend patterns ⚠ on track, drift accumulating

Good

  • Feature-first organization is in place. app/, shared/, and features/<name>/ with the api.ts/hooks.ts/types.ts/components//routes/ shape from CODING_STANDARDS.
  • App.tsx = 11 lines, pure provider / router composition.
  • TanStack Query owns server state. No useEffect-fetch antipatterns spotted.

Save/Save-As/Discard mutations need to invalidate the Rooms query cache. Today this is a direct import from features/equipment/hooks. As Windows / Envelope / Equipment-Fans / Equipment-ERVs grow their own queries, the Projects feature will need to know all their cache keys. Two options:

Inconsistent

  • Feature folder name vs. API resource name. features/project_document/ mounts routes under /api/v1/projects/{pid}/versions/{vid}/... and a separate /api/v1/projects/{pid}/diff. The folder name doesn't match the SQL table (project_versions + project_version_drafts) or the api.md group name (Versions / Drafts / Diff / Downloads). Resolve in the ADR-1 / ADR-15 split.
  • "Rooms" vs. "Equipment". The PRD USER_STORIES.md calls it the "Equipment tab" (and routes 30-tables-equipment.md as the story file), but the document table is tables.rooms. Frontend folder features/equipment/ houses Rooms-only code today. Fine, but when ERVs / Fans / Pumps land they'll all live there too — confirm that intent.
  • "Open" vs. "Make default" on versions. User-facing UI calls it "Open"; the API endpoint parameter is make_active. These are two different concepts; lumping them under one button has already caused a TB-05 review flag.

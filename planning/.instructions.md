# Planning Folder Instructions

Use this file before adding, moving, or relying on anything under
`planning/`.

When working under `planning/features/`, also read
`planning/features/.instructions.md` before starting or resuming a
feature. It captures feature-folder discovery rules and reusable
browser-work lessons that should not get lost when an individual
feature folder is archived.

## Purpose

`planning/` is the tracked working-planning surface for PH-Navigator V2.
It is the right place for feature PRDs, phase plans, progress ledgers,
review artifacts, decision logs, implementation reports, and archived
planning material that should remain available to future agents.

Do not use `planning/` for local scratch that should not be committed.
Use gitignored `working/` for that.

## Source-Of-Truth Rules

1. Stable product, architecture, stack, UI, story, and technical
   contracts live in `context/`.
2. Active feature-specific intent and execution state live in
   `planning/features/<feature>/`. Deferred v1.1 candidate features live
   in `planning/features_v1.1/<feature>/`. Longer-horizon, speculative
   features (no schedule, business-gated) live in
   `planning/features_v2.0/<feature>/` using the same folder shape.
3. Cross-cutting refactors that span many features (not one new
   feature's intent) live in `planning/refactor/<slug>/` using the same
   folder shape as a feature. Use this for consolidation / cleanup work
   driven by a code review rather than by a product story.
4. Accepted decisions from a plan or review must be folded back into the
   applicable `context/`, `planning/features/<feature>/`, or
   `planning/refactor/<slug>/` document in the same docs pass.
5. Historical dated plans stay reachable under
   `planning/archive/dated/<YYYY-MM-DD>/`; do not treat archived dated
   files as current unless a current feature `README.md` or `STATUS.md`
   points to them.
6. Code review artifacts live in `planning/code-reviews/<YYYY-MM-DD>/`
   unless a feature folder has a more specific `reviews/` directory.

## Feature Folder Contract

Every active feature folder should use this shape. Deferred v1.1
candidate folders use the same shape under `planning/features_v1.1/`.

```text
planning/features/<feature-slug>/
  README.md        # router: scope, read order, phase map
  PRD.md           # product / behavior contract
  STATUS.md        # current state, next step, blockers, verification
  PLAN.md          # optional high-level implementation sequence
  decisions.md     # optional accepted/rejected decisions
  research.md      # optional research notes / precedents
  phases/          # phase-specific implementation plans
  reviews/         # feature-specific reviews, if not dated globally
  assets/          # screenshots, diagrams, reference images
  archive/         # superseded feature-local material
```

For small or dormant features, `README.md`, `PRD.md`, and `STATUS.md`
are enough.

## Naming

- Feature folders use lowercase kebab-case: `assembly-builder`,
  `editable-fields`, `ip-si-unit-switching`.
- Canonical feature files use uppercase conventional names:
  `README.md`, `PRD.md`, `STATUS.md`, `PLAN.md`.
- Phase files live under `phases/` and use
  `phase-NN-short-title.md`, with zero padding.
- Dated archives use `planning/archive/dated/YYYY-MM-DD/`.
- New planning documents should include front matter with at least
  `DATE`, `TIME`, `STATUS`, `AUTHOR`, `SCOPE`, and `RELATED`.

## Status Vocabulary

Use these terms consistently in `STATUS.md` and phase ledgers:

- `Active` - current implementation/review focus.
- `Blocked` - cannot proceed without a named decision or external fix.
- `In review` - implementation exists; review/gates are incomplete.
- `Implemented on branch` - code exists off main; merge/verification is
  still pending.
- `Merged to main` - landed on main with evidence.
- `Complete` - landed, verified, and docs-pass folded decisions back.
- `Deferred` - intentionally out of current scope.
- `Superseded` - replaced by a newer plan or contract.

## Working Scratch

`working/` is ignored by git. Use it for local notes, branch-specific
handoffs, temporary screenshots, logs, and experiments. Promote durable
findings into `planning/` or `context/`; never leave accepted project
truth only in `working/`.

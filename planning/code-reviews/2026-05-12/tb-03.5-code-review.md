## Medium

### M1. Inconsistent query-key shapes across features

`projects` uses `["projects", "list" | "detail", id]` (key role first,
scope last). `project-status` uses `["project-status", projectId, "items"]`
(scope first, role last). Pick one before more features land.
Recommendation: copy the `projects` convention —
`["project-status", "list", projectId]` — so invalidation patterns like
`["project-status"]` and `[..., "list"]` mean the same thing across
features.

### M2. Auth-boundary invalidation is hard-coded to `["projects"]`

`useSignInMutation`/`useSignOutMutation` invalidate the project keyspace
by string literal. Today that's enough because access-sensitive UI is
gated by `project.access_mode`. The TB-04 draft-state queries will be
access-sensitive too, and a future contributor who forgets to extend this
list will reintroduce the exact bug the lesson log warns about ("public
viewer cache does not survive sign-in/sign-out"). Two reasonable shapes:

- Centralize via `queryClient.invalidateQueries({ refetchType: 'all' })`
  on auth boundary (blunt but safe), or
- Register access-sensitive query keys in one module (e.g.
  `features/auth/access-keys.ts`) and iterate it in the auth mutations.

### M3. `App.tsx` exceeds the CS hard limit (500) without an exception

Mechanically follows from H1 but worth calling out independently: the
standard says a written exception is required in the plan/review doc and
the TB-03.5 plan does not grant one.

## Low / nits

- **L1.** `features/project_status/lib.ts:48` — `formatStatusDate` is a
  one-line pass-through to `formatProjectDate`. Either drop it and import
  the shared helper directly in `StatusItemRow` and `lib.test.ts`, or
  document why the indirection exists (the test does cover it, so removing
  it costs the test one rename).
- **L2.** `useProjectQuery` (`features/projects/hooks.ts:23`) throws
  `new Error("Project ID is required.")` inside `queryFn` even though
  `enabled: Boolean(projectId)` already prevents the call. Either
  non-null-assert via narrowing earlier or drop the throw.
- **L3.** `createQueryClient()` sets `retry: false` for everything.
  Defensible for MVP; revisit before staging carries real traffic so
  transient network blips don't bubble to the user as hard errors.
- **L4.** `RequireAuth` (`App.tsx:85`) hard-codes the set of auth-failure
  `errorCode`s. If the backend renames one of
  `not_authenticated|session_expired|session_invalidated|invalid_session`,
  the redirect silently breaks. Either own this list in `features/auth/`
  or check `status === 401` only.
- **L5.** Test file is still at `frontend/src/App.test.tsx` and imports
  from `./app/App`. After H1 the tests should colocate per feature; not
  urgent.

- **CS frontend layout:** half-applied. Auth and projects routes still
  inside `App.tsx`; no `app/router.tsx`. (H1)
- **TanStack Query:** correctly adopted for server state; CS server-state
  rule is met.
- **Backend/frontend boundary, units, calculations:** untouched by this
  slice; nothing to flag.
- **Security:** no regression. Cookie-based auth, request-id propagation,
  link-sanitizer, and access-mode gating preserved.
- **Performance:** no concerns; query cache and `staleTime: 15_000` are
  reasonable defaults.

## Verdict

The slice delivers the stated server-state migration cleanly —
`QueryClientProvider`, feature-local API/types/hooks, StatusTab split,
shared date helper, helper tests. The single material issue is **H1**:
`App.tsx` is still a feature container and the CS-mandated route
extraction was the explicit reason TB-03.5 was scheduled in the first
place. Worth fixing in this slice (or as a tightly scoped TB-03.6) before
TB-04 adds another editable surface that would otherwise repeat the same
pattern. M1/M2 are inexpensive to land alongside.

1. Executive Summary

Bottom line: the core Phase 1 architecture is directionally right. The app now proves the intended through-line: relational project metadata, a Pydantic-validated ProjectDocumentV1, server-side drafts, explicit Save / Save As, public read-only project URLs, and an MCP read surface that reuses project/document services.

The single large JSONB field for saved project model data is still the right persistence move for V2 v1. The current friction is not Postgres JSONB itself; it is the lack of a generic document-table boundary now that more tables are about to arrive.

The riskiest pattern to fix before Phase 2 is that document/version chrome and draft logic are still Rooms-specific in several places. That was acceptable for a tracer bullet, but it will become architectural debt if Windows, Envelope, ERVs, Fans, and assets copy the same shape.

5. JSONB Data Model Decision

Recommendation: keep the single validated JSONB project document as the saved model source of truth for V2 v1. Do not pivot to relational project-element tables now.

Why JSONB still fits

  • Version immutability: certification submits and closed versions are naturally frozen as document bodies. A relational working-table plus snapshot layer would reintroduce the duality V2 was meant to remove.
  • Catalog bookshelf semantics: copied catalog values and catalog_origin metadata belong with the project version that owns them, not as live joins back to catalog rows.
  • Exchange format: honeybee_ph / PHX / downstream export wants a file-format-style model. The current body is already a project file, not merely persistence.
  • LLM/MCP friendliness: whole-document fetch, stable row ids, JSON schema, JSON-Patch, and table slices all line up with agent workflows.
  • MVP query scope: current planned reads are project/version scoped. There is no real cross-project query workflow yet.

Risks to manage

  • Write granularity: replacing a whole table is acceptable for Rooms, but Envelope and Windows may need semantic patch ops to avoid large conflict surfaces.
  • Validation sprawl: document validators can become a monolith unless each table owns its row model and reference validators.
  • Table-slice duplication: every new table needs read/replace/download/diff behavior. That belongs behind a registry, not copied as rooms branches.
  • Future reporting: generated columns, GIN indexes, or sidecar search may be needed later, but the plan already defines measured triggers. Do not add them preemptively.

Recommended architecture

This keeps one JSONB document while making the application boundary deep enough to support more tables.

6. Primary Findings

F1. The document service needs a generic table-slice boundary now

Evidence: backend/features/project_document/service.py is 525 lines and combines generic version/draft operations with Rooms-only operations. Examples: get_saved_rooms_slice and get_draft_rooms_slice at lines 57-105, replace_rooms_slice at lines 142-219, Rooms-only diff at lines 402-408, require_rooms_table at lines 417-423, and apply_rooms_replace / rooms_response at lines 499-525.

Why it matters: Windows, Envelope, ERVs, Fans, and assets will all need table-slice reads, draft writes, diff/download semantics, ETag checks, and validation. Copying the Rooms path will create shallow duplication and make the draft system harder to reason about.

Recommendation: split before adding the next editable table:

  • document_store.py or versions.py: load saved body, load current draft view, compute ETags, raw download escape hatch.
  • drafts.py: create/update draft, save, save-as, discard, lock checks, audit logging.
  • tables/registry.py: map table_name to a table handler.
  • tables/rooms.py: Rooms row model, option keys, slice response, replace/apply behavior.
  • diff.py: shared diff summarizer over registered tables.

F2. Project header/version chrome is coupled to Rooms

Evidence: frontend/src/features/projects/components/ProjectHeaderControls.tsx imports useRoomsSliceQuery from Equipment at line 4 and uses it at line 45 to infer hasDraft at line 47. Save uses the Rooms slice version ETag at lines 85-89. Discard copy says "Discard the current Rooms draft?" at line 109. The table download link is hard-coded to Rooms at lines 16 and 201-205.

Why it matters: Save / Save As / Discard / Diff are document-version operations, not Rooms operations. If this remains as-is, the header will either lie when another table has a draft or accumulate cross-feature imports for each table.

Recommendation: add a document-draft summary query, for example GET /api/v1/projects/{pid}/versions/{vid}/draft, returning source, version_etag, draft_etag, dirty_tables, last_patched_at, and lock state. Then make project chrome depend on that feature, not on Equipment/Rooms.

F3. The generic route names currently hide Rooms-only contracts

Evidence: backend routes expose /document/tables/{table_name} and /draft/tables/{table_name}, but both use RoomsSliceResponse and RoomsSliceReplaceRequest in backend/features/project_document/routes.py lines 58-94. Other table names are rejected by require_rooms_table.

Why it matters: callers see generic URL structure, but the response body is not generic. That is fine for a tracer bullet, but it becomes confusing once there are more table types.

Recommendation: either make the endpoint genuinely generic through a table registry or rename the tracer endpoints to Rooms-specific paths until the registry lands. The registry is preferable because it preserves the planned API shape.

F4. Read-safe-mode is under-implemented relative to the plan

Evidence: the plan calls for an unsupported-schema envelope that still permits raw JSON download. Current implementation returns invalid_project_document for table reads when schema validation fails, while /download returns raw JSON. The test test_project_download_returns_raw_body_when_schema_is_invalid verifies this behavior.

Why it matters: the raw download escape hatch is useful, but the frontend cannot render a consistent "old schema read-only" view from the table/document APIs yet.

Recommendation: make a HITL call: either redefine TB-06 read-safe-mode as "download-only recovery for Phase 1" or implement the fuller envelope before marking TB-06 complete. I recommend the fuller envelope before more table pages land.

F5. Same-editor tab handling is correct in spirit but too table-specific

Evidence: frontend/src/features/equipment/hooks.ts defines ROOMS_DRAFT_CHANNEL and Rooms-specific broadcast message types at lines 6-19, handles cache coordination at lines 67-123, and Equipment freezes active-room edits in EquipmentTab.tsx lines 34-48 and 171-177.

Why it matters: the same policy is needed for Windows, Envelope, and future MCP write leases. Repeating this per table will produce inconsistent conflict UX.

Recommendation: extract a shared draft broadcast/freeze helper keyed by {projectId, versionId, tableName, scope}. Table features should supply scope-overlap detection, not own the whole channel protocol.

7. UI And Frontend Patterns

Good pattern: the frontend is already mostly feature-first: features/auth, features/projects, features/project_status, and features/equipment each have local API/hooks/types/components/routes. App.tsx is small provider/router composition.

React best-practice alignment

  • Server data uses TanStack Query rather than manual useEffect fetches.
  • Feature hooks own query keys and mutations.
  • Route components compose feature components instead of holding raw fetch logic.
  • Tests cover helper behavior and major app flows.

Frontend friction

  • Cross-feature import in project chrome: ProjectHeaderControls imports Equipment hooks. Move document state to a project-document feature so the project header remains table-neutral.
  • Route component size: EquipmentTab.tsx is 219 lines and still reviewable, but it owns loading/error state, edit modal state, conflict state, broadcast handling, delete, save, and subtab scaffolding. Split before ERVs/Fans land.
  • Stub table: TablePrimitiveStub is intentionally small and accessible enough for the tracer, but it is not the planned DataTable. Phase 2 should extract the real controlled table API before catalog and Windows pickers depend on it.
  • Browser confirmation dialogs: native window.confirm is fine in a tracer, but this app's high-stakes save/version operations should move to typed modal flows before production editable workflows.

10. Reuse And Nomenclature

Good reuse

  • MCP read tools reuse project, status, and document service layers instead of inventing a separate read model.
  • Project access uses a single forward-compatible access seam.
  • Frontend server state is centralized through TanStack Query hooks.
  • Shared error envelope and request-id plumbing are in place.

Duplication / consolidation candidates

  • Document table read/replace/download/diff should consolidate around a table registry.
  • Draft ETag / conflict / broadcast handling should be reusable across tables.
  • Project header save state should reuse a document summary instead of reloading a table slice.
  • Single-select option lifecycle exists in Rooms helper code; this belongs in a shared table/field module when the real DataTable lands.

Nomenclature

Language is mostly consistent: project_versions, ProjectDocumentV1, draft, version_etag, draft_etag, Rooms, single_select_options. The main inconsistency is conceptual: "project document" routes are generic, but many implementations and UI labels still say Rooms. Resolve this by making the generic table boundary real.

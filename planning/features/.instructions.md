# Feature Planning Instructions

Use this file before starting, resuming, or archiving work under
`planning/features/`.

## Feature Folder Discovery

For any feature folder:

1. Read the feature `README.md` first.
2. Then read `STATUS.md`.
3. Then read the active phase file under `phases/`, if one is named.
4. Check for feature-local operational notes before browser or UI work,
   especially files named like `AGENT_*_NOTES.md`, `.instructions.md`,
   or `.index.md`.

Feature folders may later move to `planning/archive/`. Do not leave a
cross-feature lesson only inside one active feature folder if future
features need it. Promote reusable workflow guidance into this file,
`context/ENVIRONMENT.md`, `AGENTS.md`, or `CLAUDE.md` depending on
scope.

## Browser Work For Frontend Features

Use the repo-wide local UI baseline from `context/ENVIRONMENT.md`:

- Frontend: `http://localhost:5173`
- Backend: `http://localhost:8000`
- Agent login: `codex@example.com` / `password`

Do not switch to Vite fallback ports like `5174` unless backend CORS
has also been changed. Do not sign in as `ed@example.com` unless Ed
asks; PHN has a single-active-session rule and that can invalidate
Ed's own browser session.

Before browser work:

```bash
curl -i http://localhost:8000/api/v1/auth/session
```

Expected signed-out health response: `401` with `not_authenticated`.
`SESSION CHECK FAILED` or `Failed to fetch` in the browser usually
means backend down, wrong origin, or CORS mismatch.

## Known Browser Tool Limits

These are not app bugs by themselves:

- In-app Browser `fill()` / text-entry APIs can fail because they route
  through the Browser virtual clipboard setup.
- `type()` can hit the same clipboard-backed path.
- Browser evaluate may not expose `fetch`.
- Browser evaluate may not expose `XMLHttpRequest`.
- PHN session cookies are `HttpOnly`; page JavaScript cannot install or
  repair auth by setting `document.cookie`.
- Feature debug hooks attached to `window` may be inaccessible from the
  in-app Browser evaluate sandbox even when the page visibly works.

Working fallbacks:

- Use low-level DOM keypress events for simple sign-in when available.
- Use shell `curl` for API health checks instead of browser evaluate.
- Use Playwright e2e or Node Playwright for deterministic assertions,
  auth helpers, network/API setup, and page `evaluate`.
- Use the in-app Browser mainly for visual walkthroughs, DOM snapshots,
  control states, and screenshots.

## Isolated Smoke When The Dev Stack Is Busy

If Ed's backend (`:8000`) / browser are already running, do **not** take
over the ports or sign in as Ed (single-active-session rule). Stand up an
isolated copy of the *worktree's* code instead, so nothing of his is
touched (lesson from `envelope-hbjson-import`):

1. Fresh throwaway DB: `CREATE DATABASE ph_navigator_v2_wt OWNER phn`, then
   `DATABASE_URL=…/ph_navigator_v2_wt uv run alembic upgrade head` and
   `DATABASE_URL=… uv run python -m scripts.seed_user --email
   codex@example.com --password password`.
2. Backend on a free port: `DATABASE_URL=…/ph_navigator_v2_wt uv run uvicorn
   main:app --host 127.0.0.1 --port 8001`.
3. Frontend on the free `:5173` pointed at it:
   `VITE_API_BASE_URL=http://localhost:8001 pnpm run dev -- --port 5173
   --strictPort`.
4. Tear it all down afterward (kill both servers, `DROP DATABASE
   ph_navigator_v2_wt`, delete screenshots / `.playwright-mcp/`).

**Cookie same-site gotcha:** the page origin and the API base must share
the registrable domain or the `HttpOnly` auth cookie will not flow. Vite
binds `localhost` (IPv6 `::1`); set `VITE_API_BASE_URL=http://localhost:8001`
(not `127.0.0.1`) so page=`localhost:5173` and API=`localhost:8001` are
same-site (the browser still reaches a `127.0.0.1`-bound backend via
happy-eyeballs).

**File-upload flows:** a hidden `<input type=file>` triggered by a
programmatic `inputRef.click()` does **not** reliably surface a file
chooser the Playwright MCP can fill (`browser_file_upload` reports "no
modal state"). Confirm the menu item + hidden input render, then exercise
the real upload→preview→apply path via the page's own `fetch` in
`browser_evaluate` (build a `File`/`FormData`, POST the multipart preview,
then the command) — this still hits the live backend routes end-to-end.

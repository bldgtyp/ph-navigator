Accepted design decisions, folded here so the phase plans can assume them. Rationale lives in
`PRD.md`; this is the short ledger.

| # | Decision | Rationale (PRD ref) |
|---|----------|---------------------|
| D1 | **Option B — unit-aware formula fields.** A formula that returns a number may carry a display-unit; it formats through the existing unit path (SI/IP toggle, precision, suffix). | §4 — Option A ships a unit-less airflow column, failing the use-case. Ed accepted B. |
| D2 | **The output unit is a free-choice display LABEL.** Never derived from, nor validated against, the formula's inputs. No dimensional algebra. | §4a — "same unit type" is both wrong (`/0.77`) and insufficient (`W÷CFM` morphs). |
| D3 | **No unit-type validation of formula dependencies** — not even a soft warning in v1. | §6, §4a — a soft warn can't cover `/literal` or morphing formulas; an inconsistent half-measure. |
| D4 | **Reuse the existing number `units` config shape and key** (`units: {mode, unit_type, si_unit, ip_unit, precision_si, precision_ip}`) on formula fields *in the stored config*, not a new `display_units` key. Validity is gated on `result_type == "number"`. (Wire transport is different — see D12.) | Shares `validate_number_config` (backend) and `formatNumberUnitsDisplay` (frontend) verbatim; formula config is an open dict, so no collision. The result_type gate disambiguates the overload. Persisted formula configs always carry `result_type` (set_formula writes it), so the stored invariant is coherent. |
| D5 | **Unit carry-forward on conversion.** `number(units) → formula` defaults the formula's display-unit to the source field's units; `mode` carries too (a `fixed` catalog field stays `fixed`/locked; an `editable` field stays editable). Plain unit-less number → formula stays unit-less. | §4 recommendation — smallest gesture that keeps Supply Air Flow displaying as airflow. |
| D6 | **Round-trip symmetry.** `formula(units) → number` carries the display-unit back onto the resulting number field — including when the client sends no explicit units (server seeds the carry-back; the guard's effective-units rule in D13 treats that as legal). | §5.4 — a `number → formula → number` round-trip restores the original unit; built-in airflow fields can be reverted to seed config; **undo of the forward conversion works**. |
| D7 | **Units are threaded through `set_formula`, never stashed in an intermediate field config.** `TableFieldDef.validate_config` runs on every construction; a formula config carrying `units` without a `result_type == "number"` would fail validation. So `apply_set_formula` is the single construction point that reconciles `{source, ast, deps, result_type}` **and** `units`. Strengthened by D14: that reconciliation runs on *every* formula-target bundle, not only when the source changes. | `phase-02-backend.md` sites 4-5. |
| D8 | **The export EMITS computed values (in-repo) and lands FIRST — Phase 1 — so no built-in ever needs an interim lock.** Computed values are never persisted (derived-on-read — correct, keep it); MCP + frontend already surface `rows_computed`; only the gh_api tabular export drops them (`tables_export.py:19-20`, a pre-logged gap). *Revised 2026-07-09 review:* the guard relax is table-agnostic and there are **18 fixed-unit built-ins across 8 exported tables** (only Rooms even has a `field_type_locked_keys` list) — a temporary lock on just the two Rooms airflow fields would leave ~16 others exposed to silent empty-value export (energy-model corruption). Landing the export fix before the guard relax dissolves the problem for all of them at once. | §7.6, §7.16 — reordering is cheaper and safer than any lock. Phase 1 is standalone (~25-40 LOC) and needs nothing from the other phases. |
| D9 | **The change lives in shared data-table modules, not per-table code.** Every FieldDef table (`rooms`, `space_types`, all equipment tables, catalogs) inherits the behavior through the uniform `TableFieldRegistry` contract; no per-table wiring. | The schema-mutation pipeline, `custom_fields.py`, `formula/*`, and `shared/ui/data-table/*` are table-agnostic. |
| D10 | **Export wire shape = INLINE.** The gh_api export emits a formula field's computed value inline on the record keyed by `field_key` (no separate `computed` block). No collision — a formula has no stored cell. GH reads a formula column like any field via the passed-through `field_defs`. | Ed confirmed 2026-07-09. Zero client work if the GH side reads fields generically (its design goal). |
| D11 | **Formula display-unit UI = REUSE the number field's `FieldConfigSectionNumberUnits` verbatim** (full unit-type picker; supports selected / none / disabled-`fixed`), relabeled "Display units" + a one-line hint ("Formats the computed result; applies to numeric formulas"). No bespoke "keep/clear/change" control. | Ed's lean 2026-07-09. One units UX across number + formula (uniformity iron-law); the retag case (`W÷CFM → electric_efficiency`) needs the full picker anyway; least code. Section shown for formula fields; backend authoritative on dropping units for non-numeric results (D7). |
| D12 | **Display units travel on the wire as a top-level bundle-mutation field (`display_units`), never inside `after.config` for a formula target.** `EditFieldBundleMutation.after` is a `TableFieldDef`, and `validate_config` is a `model_validator` that runs **at request parse** — an `after.config = {source, units}` with no `result_type` (server-owned; the client never sends it) would 422 before `bundle.py` runs, so the originally-planned in-config carry-forward payload can never reach the bundle. This is the same reason `formula_source` is already a top-level mutation field rather than a config key — follow that precedent. Tri-state semantics: **absent** = carry forward the existing field's units (default); **explicit clear** = bare-number formula; **provided** = set/retag. The frontend strips `units` from a formula target's `after.config` unconditionally. | §7.12 — found in the 2026-07-09 review; D7's step-4/step-5 threading only covers in-bundle constructions, not wire parse. |
| D13 | **Fixed-unit fields may change type only number ⇄ formula, and the guard compares *effective* units.** The guard (one shared helper — today duplicated verbatim in `bundle.py` and `type_conversion.py`) fires when the existing field has `units.mode == "fixed"` and the **effective after-units** differ from the existing units, where effective = explicit request units if provided (config `units` for a number target, `display_units` for a formula target), else *the units the server will carry/seed* (= existing units on number⇄formula conversions and same-type formula edits). Conversions of a fixed-unit field to any **other** type (text, url, …) stay rejected — same protection as today, where the old guard blocked them incidentally. Consequences: `formula(fixed) → number` with no explicit units passes (carry-back, D6, undo); a `formula(fixed) → formula` units retag is rejected **backend-side** (not just disabled in the modal — MCP/raw API can't bypass); `number(fixed) → number` unit edits stay rejected. `editable`-units fields are unrestricted, as today. | §7.13/§7.14 — the naive "fire only when `after.field_type == number`" relax (a) blocked the reverse conversion and undo, and (b) silently dropped all backend enforcement of `fixed` once the field was a formula. |
| D14 | **Bundle step 5 (`set_formula` reconciliation) runs for EVERY formula-typed target**, not only when `formula_source` is provided. Source defaults to the existing stored source; carried units resolve per D12's tri-state. Without this, (a) a same-type formula **rename** would drop units at step 4 (whose config no longer carries them per D12), and (b) a **units-only retag** (the flagship `W÷CFM → electric_efficiency` case) would silently vanish — `handleSave` sends `formulaSource` only when the source draft is dirty, and step 5 currently short-circuits on `formula_source is None`. Making the backend re-construct unconditionally is robust for all clients (modal, MCP, raw API) instead of demanding the frontend always resend the source. Conversion *into* formula still requires an authored `formula_source` (matrix `discard_then_author`, unchanged). | §7.15 — found in the 2026-07-09 review. |

## Open (do not block Phase 2/3 on these)

- **O2 — Precision defaults** when a formula gains display-units but the source had none (only
  relevant if we later allow adding units to a from-scratch formula). Not on the conversion path.
- **O3 — Soft-hide the Display-units section for non-numeric formulas** (refinement of D11).
  Requires client-side result-type inference; v1 shows the section always and lets the backend
  reconcile. Nice-to-have, not blocking.
- **O4 — Registry-drift closure direction.** Frontend `NUMBER_UNIT_TYPES` carries `power` /
  `length_mm`, absent from backend `NUMBER_UNIT_REGISTRY`. Phase 2 makes closing this a
  prerequisite (D11 hands formulas the full picker, so it's now easy to hit). Direction —
  add the types to the backend registry vs. hide them in the frontend picker — is the
  implementer's call at Phase-2 time; either way a shared snapshot test (frontend ⊆ backend)
  locks it down.

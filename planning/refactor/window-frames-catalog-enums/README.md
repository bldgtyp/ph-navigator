# Refactor — Window-Frames Catalog → constrained vocabularies + derived name

## Why this exists

The frame-types catalog stores six categorization fields as free **text**. The
seeded data already shows the failure mode this invites: `mull_type` holds both
`OP-to-FX` and `OP-TO-FIX` for the same concept, and `manufacturer`/`brand` are
swapped on the `Mercury | CURRIES` rows. We want enforced vocabularies so values
match reliably, and we want `name` to be a **derived** label so it can never
drift from its parts.

This was always the plan: the original feature explicitly deferred it
(`planning/archive/frame-types-catalog/PRD.md:377` — *"D4. Soft-enum stays as
text-with-suggestions for v1"*). The AirTable seed has now landed, so the
follow-up is unblocked.

## What Ed asked for

1. Convert six text fields → single-select: **Manufacturer, Brand, Use,
   Operation, Location, Mull-Type**.
2. Make **Name** a formula:
   `MANUFACTURER & IF(PREFIX,…) & IF(BRAND,…) & IF(USE,…) & IF(OPERATION,…) &
   IF(LOCATION,…) & IF({MULL-TYPE},…) & IF(SUFFIX,…)`.
3. Seed the single-select options from the values already in the catalog.
4. Let users **add new option values** in the future.

## Headline conclusions

- **The formula grammar already supports Ed's exact formula** — `IF`, `&`, and
  AirTable null/empty truthiness are all implemented. No grammar change needed.
- **NAME-as-derived is lossless** against existing data — the current `name`
  values are exactly what the formula produces (they were generated by that
  AirTable formula on export).
- **The real work is the option store.** Extensible single-select options are a
  solved problem for *project-document* tables but **not for catalogs** — the
  catalog backend has no per-field option persistence today (materials `category`
  uses a frontend-declared, *locked* list). Requirement #4 needs new backend.
- **Downstream blast radius is small and mostly backend** (import/export,
  default-frame lookup, drift comparator). No frontend-consumer breakage.

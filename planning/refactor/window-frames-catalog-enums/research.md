# Research — Window-Frames Catalog enum + derived-name refactor

All claims below are cited to `file:line` at the state of 2026-06-23.

## 1. Current data shape

The frame-types catalog is a **relational table** (`catalog_frame_types`), not a
JSON document. Columns were flattened in
`backend/alembic/versions/20260604_0017_catalog_frame_types_flatten.py:35-56`.
The six target fields plus `prefix`/`suffix`/`material` are all
`sa.Text(), nullable=True`. `use`/`operation`/`location`/`mull_type` are clamped
to 40 chars in the Pydantic layer (`backend/features/catalogs/frame_types/models.py:83-86`);
`manufacturer`/`brand` allow 200.

- **`name`** is a required, user-editable column
  (`models.py:132-137`, create min_length=1 max_length=200; update patches it).
- There is **no UNIQUE** constraint on `name` — only a partial index
  `ix_catalog_frame_types_active_name` on `(name) WHERE deleted_at IS NULL`
  (`20260514_0009_catalog_frame_and_glazing.py:127-132`).
- There is **no composite uniqueness** on the
  `(manufacturer, brand, use, operation, location, mull_type)` tuple. Identity
  is the `id` UUID.

### The seed proves the motivation

`backend/seeds/catalogs/frame-types.v1.json` (≈200 rows, an
`bldgtyp:airtable-export`) already contains the exact inconsistencies Ed wants to
prevent:

- **`mull_type`**: `OP-to-FX`, `OP-to-OP`, `FX-to-FX` — **and `OP-TO-FIX`**
  (line 2808) for the same concept as `OP-to-FX`.
- **`manufacturer`/`brand` swapped**: `Curries | Mercury` (line 352) vs.
  `Mercury | CURRIES` (line 1340) — same product, fields transposed.
- **`source`** casing: `Manufacturer` vs `manufacturer` (line 3216) (not in the
  six, but the same class of error).

### Distinct values in the seed (the initial option sets)

- **use** (6): `Door, Window, Lift & Slide, Curtain Wall, Skylight, Sidelite`
  (+ null on the swapped Mercury rows).
- **operation** (7): `Inswing, Outswing, Casement, Fixed, Tilt-Turn, Sliding,
  Double-Hung`.
- **location** (6): `Head, Jamb, Sill, Mull-V, Mull-H, Any`.
- **mull_type** (3 after cleanup): `OP-to-OP, OP-to-FX, FX-to-FX` (fold
  `OP-TO-FIX` → `OP-to-FX`). Null is common and valid.
- **manufacturer** (~14): `Alpen, Curries, Internorm, Intus, Kawneer, Lamilux,
  Mercury*, Rehau, smartwin, Tishler, Viking, Wythe, Zola` (+ the `Default`
  artifact and `Mercury*` swap artifact to clean up).
- **brand** (~24): `Tyrol, Zenith, Mercury, HF510 - Home Pure, HS330, Supera,
  1600 UT CW Alu PP, 1600 UT CW FG PP, FE, CURRIES, Artevo, compact, entrance,
  sliding, Sipo Mahogany, Innova, SW14, SW17, PremiSlide, Stuyvesant,
  ThermoPlus Clad II, SDH, Thermo Wood, ThermoPlus Clad III`. **High
  cardinality, product-line-specific** — see `decisions.md` D-1.

## 2. NAME-as-formula is lossless, and the formula needs no engine change

The current `name` values are **exactly** what Ed's formula produces — they were
generated by that same AirTable formula on export. Verified against the seed:

| Row name (stored) | manufacturer / prefix / brand / use / operation / location / mull / suffix |
|---|---|
| `Zola \| French \| ThermoPlus Clad II \| Door \| Inswing \| Mull-V` | Zola / **French** / ThermoPlus Clad II / Door / Inswing / Mull-V / – / – |
| `Intus \| Supera \| Door \| Outswing \| Head \| [74-AW]` | Intus / – / Supera / Door / Outswing / Head / – / **[74-AW]** |
| `Tishler \| Sipo Mahogany \| Window \| Double-Hung \| Head \| OP-to-OP` | … / Head / **OP-to-OP** / – |
| `Mercury \| CURRIES \| Outswing \| Head` | Mercury / – / CURRIES / **null use omitted** / Outswing / Head / – / – |

So composing `name = manufacturer | prefix | brand | use | operation | location |
mull_type | suffix` (drop null/empty parts, ` | ` separator) reproduces every
existing name.

**Does the formula field need updating? No.** The DataTable formula engine
already supports Ed's formula:

- `IF(cond, then, else)` exists — `IfExprNode` in
  `frontend/src/shared/ui/data-table/lib/formula/ast.ts:58-63`, evaluated at
  `lib/formula/evaluator.ts:278`.
- **AirTable truthiness** (`evaluator.ts:110-114`): `null`/`""`/`0`/`false` are
  falsey — so `IF(PREFIX, " | " & PREFIX, "")` correctly drops empty parts.
- `&` concat (`evaluator.ts:312`) with `null → ""` parity (`:156`).
- Field refs `{Mull type}` allow spaces/hyphens.
- `context/technical-requirements/data-model.md:962` documents `&` /
  `{Display Name}` / `formula` as a first-class field type with corpus parity
  tests on both frontend and backend.

**The catch:** that formula engine and the `single_select_options` store live in
the **project-document** table machinery, *not* in catalogs (see §3). So we can
reuse the *semantics*, but a frame-types `name` formula does not "just work" the
way it does on Rooms — it has to be implemented for the catalog. See
`decisions.md` D-3 for "true formula field vs. backend-computed column".

## 3. The architecture tension (the crux)

There are two parallel field-schema worlds in this codebase:

**Project-document tables** (Rooms, Equipment/Pumps/Ventilators, Spaces) —
- Single-select options persist in `ProjectDocumentV1.single_select_options`, a
  `dict[str, list[SingleSelectOption]]` keyed by `<table_path>.<field_key>`
  (`backend/features/project_document/document.py:272-286`,
  `options.py:29-39`).
- Options are **fully user-extensible** at runtime: mint/add/remove/reorder via
  `EditOptionsMutation`/`OptionPatchOp`, validated by `validate_option_list()`
  (`options.py:73-119`, `mint_option_id()` `:162-164`); the heat-pumps service
  is a working precedent.
- Formula fields, the `(from,to)` type-conversion matrix
  (`data-model.md:1071-1095`, incl. `text → single_select = create_options`),
  and the custom-field mutate pipeline all live here.

**Catalog tables** (materials, glazing, frame_types) —
- Flat relational tables behind raw-SQL repositories. **No** per-field option
  store, **no** formula fields, **no** custom-field pipeline.
- `plan-13` (custom fields) **explicitly excluded catalogs**: D4 — *"Catalog
  tables: no custom fields in v1. Project-document tables only. Catalog
  migration is a separate later effort."*
  (`planning/archive/dated/2026-05-24/plan-13-custom-fields-overview.md:73,551`).

### How a catalog single-select works *today* (materials `category`)

`category` renders as `single_select`, but its options are **declared in the
frontend overlay and locked** (`materials/fieldDefs.ts:32,83-87` —
`locked: ["field_type","options","delete","duplicate"]`). The value persists as
a plain string in a TEXT column, constrained by a DB **CHECK** to a hardcoded
`Literal` set (`materials/models.py:31-44`,
`20260603_0015_catalog_materials_flatten.py:68-72`).

⇒ Materials proves a catalog *can* show a single-select, but the current pattern
is **fixed/non-extensible**. Ed's requirement #4 (users add new values that
persist and are shared) is **not** met by this pattern and needs new backend.
See `decisions.md` D-2 for the storage options.

## 4. Downstream-consumer blast radius

Mapped every reader of `name` and the six fields. **No frontend-consumer
breakage; the work is backend (import/export, default-frame lookup, drift).**

### `name` as a join/match key — exactly one site

- **Default-frame lookup** — `backend/features/project_document/apertures/default_refs.py:104`
  does `SELECT … WHERE name = %(name)s …` with
  `APERTURE_DEFAULT_FRAME_NAME = "PHN-Default-Frame"`
  (`envelope_models.py:29`). The sentinel row is seeded by
  `20260605_0018_apertures_default_catalog_seed.py:44-58` with **id
  `recPHNDefFrame001` and all six component fields NULL**.
  ⇒ A formula-derived `name` would compute to **empty** for this row → the
  lookup breaks. **Fix: resolve the default frame/glazing by their deterministic
  ids (`recPHNDefFrame001` / `recPHNDefGlazng01`), not by name.** Clean and
  contained.

Everywhere else `name` is **display-only**, not a key:
- Picker filters on `location`/`operation`/`use`/`manufacturers`, never `name`
  (`backend/features/catalogs/frame_types/repository.py:93-133`;
  `frontend/.../apertures/hooks/useFrameCatalog.ts`).
- Refs aggregation dedupes on `catalog_record_id`, not `name`
  (`frontend/.../apertures/lib/refsAggregation.ts`).
- FrameRef copies `name` for display (`apertures/ref-builders.ts`,
  `apertures/types.ts` `name: string`) — keeps working; it just reads the
  computed value.

### `operation` string-matching — enum migration *fixes latent bugs*

`operation-frame-match.ts` warns when an element's operation ≠ the frame's
operation by comparing `normalize(value) = value.trim().toLowerCase()`. Today a
free-text `operation` can be `Tilt-Turn` vs `tilt-turn ` and mis-warn. Forcing
`operation` to a single-select makes this comparison **deterministic** — the
enum migration removes a real class of false/again-missed warnings. (Note: the
match is on `operation` only, not `name`.)

### Import / export (the main backend work)

- `name` is an **optional** import field
  (`backend/features/catalogs/frame_types/import_export/file_format.py:26`).
- Coerce has an `ERR_MISSING_NAME` gate and a 200-char check
  (`import_export/coerce.py:29,100-107`); `name` is in `_CANONICAL_FIELDS`
  (`:71`).
- To make `name` derived: drop the missing-name error, **compute** `name` from
  the six+prefix/suffix after coercion, bump `schema_version` (currently 1) and
  add an **upgrade step** that folds legacy/typo values into canonical options
  (this is where `OP-TO-FIX → OP-to-FX` gets normalized on import). The six enum
  fields also need **option-resolution** on import (map incoming label → known
  option; unknown → either auto-add or flag).

### Drift comparator

`backend/features/aperture_drift/comparator.py:17-22` includes `"name"` in
`_FRAME_KEYS` *"because the catalog Name can be renamed in place (TB-09.a)"*.
Once `name` is derived it can never drift independently → **drop `"name"` from
`_FRAME_KEYS`** (cosmetic; fewer/cleaner deltas).

### Other readers — no change

MCP (`apertures_mcp`, catalogs), `aperture_u_value`, `aperture_hbjson_export`,
and seeds read the fields as plain strings; enum values are still strings, so
they keep working.

## 5. Reuse inventory (what already exists to lean on)

- ✅ `SingleSelectOption` model (`project_document/rows.py:24-30`) — id/label/
  color/order; reusable as-is.
- ✅ Option helpers (`project_document/options.py`): `option_list_key`,
  `read/replace_option_list`, `validate_option_list`, `mint_option_id`.
- ✅ Frontend single-select UX: `SingleSelectCell` / `SingleSelectPopover`
  (`context/CODING_STANDARDS.md:401`), `OptionListDelta` / `newOptions` /
  `legacyOptions` schema-mutation in `data-table/types.ts` already support
  add-option-inline.
- ✅ Formula engine (frontend + backend, corpus-parity tested) — semantics for
  the derived name, *if* we choose the true-formula route (D-3).
- ✅ `text → single_select = create_options` conversion semantics already
  specified (`data-model.md:1086`) — a template for the seed/upgrade step even
  though catalogs don't run that pipeline.

## 6. What is genuinely missing (new build)

- A **catalog-scoped option store** + repository + service mutations + routes +
  Pydantic models (catalogs are global, so options are global — not per project
  document). See `decisions.md` D-2.
- **Server-side name derivation** for create/import/patch + read-only `name` in
  the UI; reject inbound `name`.
- **Default-frame/glazing resolution by id** instead of name.
- **Import schema v2** with upgrade + option-resolution.
- **Data-quality migration** of existing rows (fold typos, fix the Mercury/
  CURRIES swap, decide on the `Default`/`Mercury` artifacts).
